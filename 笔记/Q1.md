# 基于 RAG 的私有化知识仓库问答 Agent 系统 
    项目地址：https://github.com/dazzlingwuming/RAG_self.git
## 项目背景：
- 本项目是一款轻量化、可私有化部署的检索增强生成（RAG）知识问答系统，核心解决通用大语言模型（LLM）在私有 / 领域知识库场景下的核心问题：「无依据幻觉生成」「回答与知识库脱节」。
    1. 解决的核心问题：让 LLM 严格基于用户上传的私有知识库生成回答，同时具备多轮对话、复杂问题拆解、检索策略自适应的能力；
    2. 使用场景：① 个人知识库问答（如笔记、论文、技术文档的精准检索与答疑）；② 中小团队内部领域知识问答（如业务手册、接口文档的智能解答）；③ 轻量化客服场景（如产品使用手册的自动答疑）；

## 个人工作：
| 模块 | 	具体工作                                                                     | 
|------|---------------------------------------------------------------------------|
| 架构设计 | 文档预处理→向量存储→Agent 控制→检索增强→LLM 生成→web接口输出                                   | 
| Agent 层 | 实现问题解析，复杂问题拆解利用到数据检索、多轮上下文管理、检索策略切换、回答校验的核心逻辑                             | 
|数据工程| 基于 LangChain + Pandas 实现多格式文档（PDF/Markdown/TXT）的加载、清洗、语义分段                |

## Agent 目标
- 上下文管理：维护多轮对话上下文，保证连续问答的连贯性；
- 检索策略控制：根据问题进行知识检索；
- 回答校验：利用提示词确保 LLM 回答仅来源于知识库，过滤无依据的幻觉内容；
- 结果格式化：输出「核心回答 + 引用来源 」的结构化结果。

## 方案与流程：
- 文档接入层（多格式上传/清洗）→ 向量存储层（ChromaDB 向量库）→ Agent 控制层（核心逻辑）→ LLM 生成层（DeepSeek）→ 接口层（FastAPI）
  1. 文档预处理：上传文档 → Agent 调用文档处理模块 → 文本清洗（去冗余 / 格式标准化）→ 语义分段（200-500 字符 / 段，保证语义完整）→ (bge-base-zh-v1.5模型 向量编码 → 存入 ChromaDB 向量库。
  2. 问答阶段 ：用户提问 → Agent 解析问题→ 检索调用 ChromaDB 粗检索（返回 top-k 文档）→ bge-reranker-base 细检索重排序（返回 top-n 文档）→ 构建提示词（包含问题 + 检索结果）→ LLM 生成回答 → 回答校验（确保仅基于知识库内容）→ 格式化输出（核心回答 + 引用来源）。

## Prompt：
- 最关键的模板是：你是一个专业的中文问答助手，擅长基于提供的资料回答问题。
请仅根据以下背景资料以及历史消息回答问题，如无法找到答案，请直接回答“未在知识库里面找到相应内容，以下回答是基于自身理解回答”。
背景资料：{context}
历史消息：[{history}]
问题：{query}
请按照以下格式回答：
核心回答：[针对用户问题的精准回答，语言简洁明了]
引用来源：[列出回答所依据的背景资料或者历史记录中最关键片段内容”]
- 原因：
  1. 强制回答仅基于给定的知识库内容和历史对话，杜绝模型编造或使用外部知识，也可以运行模型自己回答，但是必须要先说明这次回答是基于自身知识回答；
  2. 利用历史记录解决大模型 “上下文遗忘” 问题，让多轮对话有记忆，符合真实交互场景；
  3. 通过固定输出格式，明确回答的来源，确保每一个结论都能对应到具体的知识库片段或历史对话，便于校验。

## 个人经验：
- 纯 LLM 直接调用的主要缺陷：
  1. LLM 可能会编造不存在的内容，导致回答不可靠；
  2. LLM 的回答与用户想要的私有知识库内容不匹配，无法满足用户需求；
  3. 在多轮对话中，LLM 可能会忘记之前的对话内容，导致回答不连贯。
- 优化与增强 Agent 的可靠性与可控性：
  1. 强制回答仅基于给定的知识库内容和历史对话，杜绝模型编造，但是因为可以给用户一个参考的回答，我感觉还是需要允许基于自身的知识来回答，但是需要说明；
  2. 利用历史记录解决大模型 “上下文遗忘” 问题，让多轮对话有记忆，符合真实交互场景，短窗口用于近几轮对话，但是对于超多轮和超长的对话，其实可以采取类似知识库的存储或者提取摘要
如果是存入知识库就使用模型来提取相关的历史记录，如果是摘要的话，也可以就简单存储，然后使用重排序模型需选择；
  3. 通过固定输出格式，明确回答的来源，确保每一个结论都能对应到具体的知识库片段或历史对话，便于校验，同时也可以让用户更清楚地知道回答的依据，提升用户对系统的信任度。

# 量化证据清单
## 关键配置项及取值
- 文本分段长度：250 字符或者是整体一段；
- 向量检索相似度阈值：0.7；
- 历史记录窗口大小：近 10 条对话；
- 数据向量库：ChromaDB；
- 嵌入模型：bge-base-zh-v1.5；
- 粗检索返回条数：top-20；
- 细检索返回条数：top-10；
- 大模型温度参数：0.1；
- 细检索重排序模型：bge-reranker-base；
- LLM 生成模型：DeepSeek_v2.5；
- 接口框架：FastAPI；
## 版本信息
- Python 版本：3.10；
- ChromaDB 版本：1.4.1；
- fastapi  0.128.0;
- langchain 1.2.6
